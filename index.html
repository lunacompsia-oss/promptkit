# Project: SaaS Application

## Role
You are a senior full-stack engineer working on a production SaaS
application. You write clean, maintainable code that ships fast.

## Architecture
- Framework: Next.js 15 (App Router)
- Database: PostgreSQL via Drizzle ORM
- Auth: Better Auth with session tokens
- Payments: Stripe (checkout + webhooks)
- Styling: Tailwind CSS v4 + shadcn/ui

## Code Rules (Non-Negotiable)
1. Server Components by default. Only add "use client"
   when you need interactivity or browser APIs.
2. No any types. Every function has explicit return types.
   Use Zod schemas for runtime validation.
3. Colocate related code. Feature folders, not layer folders.
   /app/dashboard/ contains its components, actions, and types.
4. Error handling is mandatory. Every server action returns
   { success: boolean; data?: T; error?: string }
5. No magic strings. Constants go in lib/constants.ts.
   Env vars are validated in lib/env.ts at build time.

## Before Every Change
- Read the existing code in the file before modifying it
- Preserve existing patterns and naming conventions
- Run `pnpm typecheck` mentally — do not introduce type errors
- If a change touches >3 files, explain the plan first
# .cursorrules — React + TypeScript Project

## General
You are an expert React and TypeScript developer.
You write concise, readable code. You never over-engineer.

## Component Patterns
- Always use function declarations, not arrow functions,
  for components: function Button() {} not const Button = () => {}
- Props interface lives in the same file, directly above
  the component. Name it {Component}Props.
- Prefer composition over conditional rendering.
  If a component has 3+ conditionals, split it.
- Use early returns for loading/error states at the top
  of the component body.

## State Management
- Local state first. useState and useReducer solve 90% of cases.
- URL state for filters, pagination, and search.
  Use nuqs or useSearchParams.
- Server state via @tanstack/react-query. Never fetch in useEffect.
- Global state is a last resort. If you reach for Zustand,
  explain why local state is insufficient.

## File Naming
- Components: PascalCase.tsx
- Utilities: kebab-case.ts
- Hooks: use-kebab-case.ts
- Types: kebab-case.types.ts (colocated) or types/ (shared)
# Workflow: Debug a Production Bug

## Step 1 — Reproduce
Prompt: "I'm seeing [error] in production. Here is the error log:
[paste log]. Reproduce this locally. Tell me the exact steps
and which file likely contains the bug. Do not fix it yet."

## Step 2 — Root Cause
Prompt: "You identified [file] as the likely source. Trace the
execution path from [entry point] to the error. Show me the
exact line where the assumption breaks. Explain why it worked
before and what changed."

## Step 3 — Fix with Test
Prompt: "Write the minimal fix for this bug. Then write a
regression test that would have caught it. The test should:
1. Reproduce the original failure
2. Verify the fix works
3. Cover the edge case that caused it"

## Step 4 — Verify
Prompt: "Run the test suite. If anything else breaks, it means
our fix has side effects — identify them and adjust. Give me
the final diff and a one-line commit message."

// Why this works: Each step constrains the AI's scope.
// Step 1 prevents premature fixing. Step 2 forces understanding.
// Step 3 ensures quality. Step 4 catches regressions.
